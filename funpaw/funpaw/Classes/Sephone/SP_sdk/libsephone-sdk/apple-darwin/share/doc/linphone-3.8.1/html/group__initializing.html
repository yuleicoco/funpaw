int sephone_core_migrate_to_multi_transport ( SephoneCore *  lc)

Migrate configuration so that all SIP transports are enabled. Versions of sephone < 3.7 did not support using multiple SIP transport simultaneously. This function helps application to migrate the configuration so that all transports are enabled. Existing proxy configuration are added a transport parameter so that they continue using the unique transport that was set previously. This function must be used just after creating the core, before any call to sephone_core_iterate()

Parameters
lcthe sephone core
Returns
1 if migration was done, 0 if not done because unnecessary or already done, -1 in case of error.

Instanciates a SephoneCore object.

The SephoneCore object is the primary handle for doing all phone actions. It should be unique within your application.

Parameters
vtablea SephoneCoreVTable structure holding your application callbacks
config_patha path to a config file. If it does not exists it will be created. The config file is used to store all settings, call logs, friends, proxies... so that all these settings become persistent over the life of the SephoneCore object. It is allowed to set a NULL config file. In that case SephoneCore will not store any settings.
factory_config_patha path to a read-only config file that can be used to to store hard-coded preference such as proxy settings or internal preferences. The settings in this factory file always override the one in the normal config file. It is OPTIONAL, use NULL if unneeded.
userdataan opaque user pointer that can be retrieved at any time (for example in callbacks) using sephone_core_get_user_data().

void sephone_core_iterate ( SephoneCore *  lc)

Main loop function. It is crucial that your application call it periodically.

sephone_core_iterate() performs various backgrounds tasks:

  • receiving of SIP messages
  • handles timers and timeout
  • performs registration to proxies
  • authentication retries The application MUST call this function periodically, in its main loop. Be careful that this function must be called from the same thread as other libsephone methods. If it is not the case make sure all libsephone calls are serialized with a mutex.
